# ADR : Utilisation d'Apache Cassandra comme base de données distribuée

## Statut
**Accepté** - Décision prise dans le cadre de la migration vers une architecture entièrement distribuée dans le cloud.

## Contexte
L'application Store Manager a évolué d'un monolithe vers une architecture microservices event-driven. Actuellement, chaque service possède sa propre base de données et la synchronisation entre ces bases est gérée manuellement. 

Avec la migration prévue vers le cloud, nous devons préparer l'application pour un fonctionnement entièrement distribué. Les défis identifiés incluent :
- Gestion manuelle complexe de la synchronisation entre multiples bases de données
- Risques de cohérence des données entre les services
- Difficulté à garantir la haute disponibilité et la résilience
- Scalabilité limitée avec l'approche actuelle

Nous avons besoin d'une solution de base de données qui assure nativement la distribution et la synchronisation des données à travers un cluster distribué.

## Décision
Nous adoptons **Apache Cassandra** comme système de base de données distribuée pour l'application Store Manager.

**Justification :**
- **Distribution native** : Cassandra gère automatiquement la réplication et la distribution des données à travers le cluster, éliminant la nécessité de synchronisation manuelle
- **Haute disponibilité** : Architecture masterless (sans point de défaillance unique) avec réplication configurable garantissant la continuité de service
- **Scalabilité horizontale** : Permet d'ajouter facilement des nœuds pour augmenter la capacité sans interruption de service
- **Modèle de cohérence ajustable** : Offre un contrôle fin sur le compromis entre cohérence et disponibilité (théorème CAP)
- **Préparation cloud** : Architecture distribuée alignée avec les exigences d'un déploiement cloud multi-régions
- **Simplicité opérationnelle** : Réduit la complexité de gestion en centralisant la logique de distribution au niveau de la base de données plutôt qu'au niveau applicatif
